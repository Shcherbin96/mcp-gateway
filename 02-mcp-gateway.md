# Проект 2: MCP Gateway

> Второй проект портфолио для fulltime remote AI Agent роли.
> Цель — показать что я думаю как production-инженер и понимаю инфраструктуру
> для безопасной работы AI-агентов в проде.

---

## Краткое описание

MCP-сервер, работающий как защищённый шлюз между AI-агентами и реальными системами компании.
Каждый tool call проходит через слои контроля: проверка авторизации, политика доступа, human-in-the-loop approval на критичных действиях, иммутабельный аудит-журнал.

Это инфраструктурный проект.
Любая компания, которая хочет внедрить AI-агентов в реальную работу, упирается в вопрос «как сделать безопасно». Большинство фрилансеров умеют только строить агентов. Те, кто умеет строить инфраструктуру вокруг них — стоят дороже.

---

## Аналогия

MCP Gateway = охранник на входе в офис.

Есть боевые системы компании: база клиентов, склад, банковские операции.
AI-агенту нужно с ними работать. Без охранника он идёт куда хочет и делает что хочет — это страшно.

Gateway проверяет пропуск, звонит начальнику на крупных операциях, ведёт журнал всех действий.

---

## Что Gateway делает технически

Это HTTP-сервер, реализующий MCP protocol.

- **Снаружи для Claude:** выглядит как обычный MCP-сервер с набором tools (`get_customer`, `update_order`, `refund_payment`).
- **Внутри:** каждый tool call проходит через цепочку из 5 слоёв контроля.
- **Снизу:** обычные API-вызовы к реальным системам компании.

---

## Архитектура — 5 слоёв на каждый tool call

```
Claude → [1. Authenticate] → [2. Authorize] → [3. Approve?] → [4. Execute] → [5. Log] → Response
```

### Слой 1: Authenticate (кто пришёл)
Проверяем токен через **OAuth 2.1** — стандарт для MCP в 2026.
Если токен невалидный, истёк, не от того клиента — отказ сразу.

### Слой 2: Authorize (что ему можно)
У каждого agent есть роль с правами.
Например: `support_agent` может вызвать `get_customer`, `update_order` — но не `refund_payment`.
Политики конфигурируются в YAML-файле, не хардкодом.

### Слой 3: Approve (нужно ли подтверждение человека)
Часть tool calls помечена как **destructive** — потенциально опасные.
На такие Gateway не отвечает Claude сразу.

**Что происходит:**
1. Claude вызывает `refund_payment(customer_id, amount=50000)`
2. Gateway видит — destructive
3. Шлёт push-уведомление мне (на телефон / Slack): «Agent X хочет вернуть 50 000 ₽ клиенту Y. Approve / Reject?»
4. Я жму approve → Gateway выполняет
5. Я жму reject → Claude получает «not approved», обрабатывает graceful

Технически: pending request пишется в Postgres, websocket доставляет уведомление.
Таймаут — например 5 минут, после этого автоматический reject.

### Слой 4: Execute (выполняем)
Реальный API-вызов к системе компании.
Обработка ошибок, retry на сетевых, нормализация ответа.

### Слой 5: Log (запись)
Каждый tool call пишется в **иммутабельный audit log**:

- timestamp
- кто вызвал (agent ID)
- какой tool
- какие параметры
- результат
- был ли approval, кто approved, когда

Журнал **write-only** — нельзя задним числом изменить запись.

**Зачем именно иммутабельный.** Compliance и forensics.
Если агент сделал что-то странное — есть лог, можно разобраться.
Если регулятор спросит «кто принял это решение» — есть запись.

## Стек

| Слой | Инструмент | Зачем |
|---|---|---|
| MCP-фреймворк | FastMCP (Python) | Готовая обвязка для transport, schema validation |
| Web-сервер | Starlette / FastAPI | Под FastMCP |
| OAuth 2.1 | Authlib или аналог | Стандартная имплементация |
| База данных | Postgres | Политики, pending approvals, audit log |
| Уведомления | Slack webhook + Telegram bot | Push для approval |
| Деплой | Railway или Fly.io | Публичный URL для демо |
| MCP transport | Streamable HTTP | Работает с Claude Desktop, Claude Code, Cursor |

Конкретные версии уточняются при сборке.

---

## Демо-сценарий

На 30-секундном видео показываем:

1. Подключаю Gateway к Claude Desktop (виден config.json)
2. Прошу Claude: «верни 50 000 рублей клиенту Иванову за заказ #1234»
3. Claude делает tool call → у меня на телефоне всплывает approve-уведомление
4. На сплит-скрине показываю: жму **reject**
5. Claude в чате отвечает: «не смог выполнить, операция отклонена администратором»
6. Открываю audit log в браузере — видна полная запись с rejector ID и timestamp

## Связь с первым проектом

Gateway можно (и нужно) переиспользовать в первом проекте.
Когда Technical Research Agent захочет вызвать какой-то external tool — он делает это через свой Gateway.
В демо это можно показать как единую систему: «вот мой агент, вот моя инфраструктура».

Это превращает портфолио из «два независимых проекта» в **единую coherent platform**.
Сильный нарратив для интервью.
